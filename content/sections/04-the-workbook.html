04 · the workbook

A workbook, in one example

The page you're looking at is a workbook. Right now there is an agent reading this section's source the way you are reading the rendered page — a few dozen lines of HTML in content/sections/04-the-workbook.html, a row in content/sections.json, and a small bit of CSS. A person could open the same file in any text editor and edit it. The agent did not have a build pipeline or a staging server between writing the file and the page changing. It wrote a file, published it, and the runtime served the new content.

That is the part the rest of the site does not show. A workbook is a place where a person and an agent can both work, in the same file, on the same artifact, with no translation layer between them. The disagreement — "this paragraph is too long", "this section is missing", "this link is broken" — is resolved by editing the file. There is no API to call, no schema to respect, no platform to ask. The edit IS the change. The panel on the right is the receipt: the next commit from this run will show up there within minutes. The FAQ answers the meta-questions about the loop.

The same loop works on any workbook, made for any audience. Open the file, read it like a document, change the part that needs changing, save it, and the runtime takes it from there. Whoever opens the file next — a person, an agent, an auditor with a different opinion — reads the same thing and starts from the same place. The file is the medium, the medium is plain, and plain is what makes the loop work.
