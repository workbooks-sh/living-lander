#+TITLE: design — the site's design system (Waldo: consult before building, reuse always)

Never invent ad-hoc styles. Every visual decision below already exists in
index.html's <style> — reuse the classes and variables; do not duplicate them
inline. If a pattern you need is missing here, the answer is the closest
existing pattern, not a new one.

* tokens (CSS variables, already defined in :root)

  --bg #0c0d10 (page) · --panel #14161b (cards) · --line #262a32 (hairlines)
  --ink #eceef2 (headings/strong) · --dim #8b909a (body) · --faint #565b64 (meta)
  --live #3fe081 (THE accent — agent/live/links; use sparingly)
  #5aa7ff (blue, runtime/blog) · #f5b945 (amber, toolkits/audit) · #b48cff (violet, dreams)
  --sans Geist · --mono Geist Mono (labels, meta, code) · --r 10px radius

* section anatomy (the only way to add a section)

  THE PAGE IS ONE DESIGNED OBJECT. Every section — human or yours — follows
  the same band rhythm, including the numbered kicker sequence (01 · the
  stack and 02 · in the wild are human; yours continue 03, 04, …, in DOM
  order). Whoever adds or moves a section renumbers the kickers after it.

  A section is ONE HTML PARTIAL: `content/sections/NN-slug.html` plus a row
  in `content/sections.json` (NN = page order; yours start at 04; an faq row
  carries `"pin":"last"`). The live page loads partials at runtime — no build,
  no compiled output to edit. The partial is exactly one <section class="grown"> with
    1. <div class="kicker">0N · <two-word label></div>  (the sequence above)
    2. an <h2> (sentence case, ≤6 words, no period)
    3. prose <p>s — var(--dim), 15.5px, ≤65ch, 2-4 paragraphs MAX
    4. optionally ONE pattern from below. Never two patterns in one section.
  Inherited styles handle width/borders/spacing — add NO inline styles except
  small one-off spacing. Sections are PREPENDED (newest first) inside #grown,
  with ONE exception: an faq section always sits LAST in #grown — FAQs live
  at the bottom of pages, always.

* patterns (pick one, copy its existing markup shape)

  - prose: h2 + paragraphs. The default. Most sections should be this.
  - faq: <details><summary> rows (see the existing FAQ section).
  - linklist: a <p> per item with an <a> + a --faint meta span (see the
    "from the agent's desk" section).
  - cards: only if genuinely enumerable; mirror .lcard markup (dark card,
    colored label, NO solid color backgrounds — color arrives as accents).

* blog post skeleton

  Copy blog/this-website-builds-itself.html's structure verbatim: head (same
  fonts), .topnote (← home), .meta (date · author · one green chip), h1,
  .article body, .signoff (avatar W + name Waldo). Include
  <script src="/assets/site.js" defer></script> before </body>. No hand-rolled
  nav. Dates are 2026.

* voice

  Plain, present tense, first person allowed (you are Waldo and may say so).
  Honest pre-launch: no users, no metrics, no praise. Sell the ecosystem
  (workbooks / runtime / toolkits), never "one file" trivia, never "sites that
  run themselves", never "software you can own". Sentence-case headings.
  Short sentences beat long ones. One idea per section.

* the one-idea law (anti-repetition — checked every audit)

  The shell already tells the pitch: 01 names the three layers, 02 tells the
  person→team→world ladder, the hero sells the ecosystem. Those beats —
  "interface, logic, and data travel as one unit", "no sign-up, starts on your
  machine", "the same artifact scales" — are TOLD ONCE, in the shell, and are
  BANNED from grown sections and banned from appearing twice anywhere. A grown
  section earns its place with ONE idea the page does not already have, stated
  in its own words. Before writing one, list each existing section's single
  idea; if yours collides, improve that section instead. An audit that finds
  the same beat told twice cuts the second telling.

  Comparisons live in blog/ — the landing page never carries a prose twin of
  a post (the agent's-desk listing already links every post). If a comparison
  earns a page presence, it is a compact TABLE (rows of facts), not prose.

* the band design law (compose, don't drip)

  The grown band is DESIGNED AS A WHOLE before it is built. In a PLAN run:
  decide what the page is missing as a set — 2 to 4 sections, each named with
  its one idea, its order, and why a first-time reader needs it — and write
  that design into the board as an objective whose child tasks are the
  sections. ADD runs then fill slots from the design; no section ships without
  a slot. When the page's needs change, redesign the band in the next plan
  run — never bolt a section onto the end because a run needed something to do.
